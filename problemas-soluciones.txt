Problema
- La app quedaba en carga continua en localhost:3000 y el servidor de desarrollo registraba panics de Turbopack.

Causa raiz
- Next/Turbopack estaba resolviendo el workspace desde archivos package.json/package-lock.json ubicados en C:\Users\Jesús Andrés en lugar de limitarse al directorio real de la app.
- Eso provocaba el error "Next.js package not found" al generar endpoints como /(dashboard)/engineering/portfolio/page.

Solucion aplicada
- Se aislaron de forma reversible los archivos externos:
  - C:\Users\Jesús Andrés\package.json -> package.json.codex-bak-20260328_1629
  - C:\Users\Jesús Andrés\package-lock.json -> package-lock.json.codex-bak-20260328_1629
- Se mantuvo la configuracion de Next con `turbopack.root = path.resolve(__dirname)` en next.config.ts.
- Para estabilizar el arranque de desarrollo se levantó la app con webpack:
  - npm.cmd run dev -- --webpack --port 3000

Verificacion
- /login responde HTTP 200.
- /clients responde HTTP 307 (redireccion de autenticacion esperable sin sesion).
- /engineering/portfolio responde HTTP 307 (redireccion de autenticacion esperable sin sesion).
- El log nuevo de error .codex-dev-3000-webpack.err.log queda vacio tras el arranque limpio.
- El log de salida confirma:
  - Next.js 16.2.1 (webpack)
  - Ready in 358ms

Notas y riesgos
- El servidor actual queda corriendo como proceso de desarrollo en el puerto 3000.
- Si se restauran los archivos C:\Users\Jesús Andrés\package.json y package-lock.json sin otra medida de aislamiento, puede reaparecer el problema.
- Si en el futuro se quiere volver a Turbopack, primero hay que resolver definitivamente la deteccion de workspace externo.
