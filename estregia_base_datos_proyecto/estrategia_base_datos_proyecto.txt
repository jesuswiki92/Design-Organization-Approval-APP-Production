Estrategia base de datos para acelerar quotations con precedentes técnicos

Objetivo
La estrategia no consiste en crear una base de datos por proyecto. El objetivo es construir una arquitectura común que permita localizar rápidamente precedentes técnicos, evaluar similitudes reales y reutilizar conocimiento validado para acelerar quotations con criterio técnico.

Principio de diseño
No se crea una base por proyecto. Cada proyecto debe existir como un expediente lógico dentro de una plataforma común. Esa plataforma debe permitir contar precedentes, comparar casos y recuperar información técnica útil sin duplicar estructuras ni aislar el conocimiento.

Bases lógicas principales
1. project_registry
Registro maestro de proyectos.
Guarda la identidad básica de cada proyecto: número, cliente, aeronave, modelo, variante, tipo de trabajo, clasificación, cert basis, estado y relación con ofertas o quotations.

2. technical_precedents
Base central de reutilización técnica.
Guarda lo que realmente se hizo en cada proyecto validado: tipo de instalación, zona afectada, ATA, sistema implicado, análisis realizados, entregables emitidos, ruta de aprobación y resultado final.

3. controlled_documents
Base de trazabilidad documental.
Guarda la evidencia que respalda cada precedente: MDL, documentos, revisiones, drawings, baseline, firmas, rutas, estados y documentos obsoletos.

4. similarity_ai_index
Base de similitud y búsqueda semántica.
Guarda resúmenes validados, chunks, embeddings y relaciones de similitud entre proyectos y documentos para encontrar casos parecidos aunque no usen exactamente la misma terminología.

Capa derivada de apoyo a quotation
quotation_support_layer
No debe ser la base fuente. Debe ser una capa derivada que combine filtros técnicos, conteo de precedentes y resultados semánticos para responder rápido a preguntas de quotation.

Ejemplo práctico: rack installation
Si entra una solicitud sobre rack installation, el sistema debe poder:
- contar cuántos proyectos similares se han hecho
- identificar cuáles fueron realmente parecidos
- revisar qué análisis técnicos hicieron falta
- ver qué documentos salieron
- detectar riesgos repetidos
- apoyar al equipo de quotation con precedentes concretos y no con texto genérico

Taxonomía técnica canónica
La estrategia depende de una taxonomía técnica única y normalizada. Si rack installation, equipment rack, avionics rack o tray installation se registran como conceptos distintos sin relación, la búsqueda será débil y el conteo de precedentes será incorrecto.

Por eso cada proyecto debe etiquetarse con términos canónicos:
- tipo de trabajo
- subtipo técnico
- aeronave, modelo y variante
- ATA o sistema afectado
- zona de instalación
- clasificación
- cert basis
- impactos técnicos
- entregables
- resultado final

Cómo se alimenta la información
1. En el alta del proyecto se crea la ficha en project_registry.
2. Durante el trabajo se registran hechos técnicos en technical_precedents.
3. Los documentos que respaldan el trabajo se guardan en controlled_documents.
4. Al cerrar o validar el proyecto, un ingeniero confirma qué conocimiento queda reutilizable.
5. Solo lo validado pasa a similarity_ai_index y a quotation_support_layer.

Ventaja estratégica de una base vectorial
La experiencia con bases de datos vectoriales puede potenciar mucho esta estrategia porque permite buscar por similitud real, no solo por coincidencia exacta de campos.

Esto es importante porque en quotations muchas veces la pregunta no llega con la nomenclatura perfecta. Un usuario puede pedir rack installation, pero los proyectos anteriores pueden estar etiquetados como equipment rack, avionics rack, tray assembly o cabinet installation. Una base vectorial permite encontrar esos casos cercanos aunque el texto no coincida literalmente.

Ventajas concretas de una base vectorial:
- mejora la búsqueda semántica de precedentes
- detecta proyectos similares aunque cambie la redacción
- prioriza ejemplos técnicamente cercanos
- acelera la revisión inicial de quotation
- ayuda a reutilizar wording, análisis y documentación ya aprobados
- reduce el riesgo de perder precedentes por diferencias de vocabulario

Conclusión operativa
La estrategia correcta no es almacenar proyectos como compartimentos aislados. Es convertir cada proyecto en un precedente técnico estructurado, trazable y buscable. La combinación de taxonomía canónica, control documental y búsqueda vectorial es la que permite acelerar quotations con datos de calidad y con criterio técnico real.
